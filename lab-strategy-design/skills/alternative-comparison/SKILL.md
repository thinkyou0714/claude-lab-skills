## Purpose

「最初に思いついた案」に流される意思決定を防ぐ。
選択肢を同じ軸で並べ、何を得て何を諦めるかを明示して、人間が選べる状態にする。

## Use When

- 戦略・施策の方向性が複数あるとき
- 「この案でいく」と決める前の最終確認
- 投資・撤退の判断を伴うとき
- 関係者で意見が割れているとき

## Inputs

以下を準備すること。不足している場合は推測せず、不足を明示する。

- **選択肢**: 比較したい案（最低2案）
- **評価したい目的**: 何を最大化／最小化したいか
- **制約**: 予算・期限・体制の制約
- **比較軸の希望**: 重視したい観点（あれば）

## Output Contract

以下の順で出力すること。順序を変えない。

1. **論点**: 選択を分ける最も重要な軸は何か
2. **根拠**: その軸を重視する理由
3. **比較表**: 案 × 軸（コスト・効果・リスク・撤退容易性・前提）
4. **含意**: 各案を選んだ場合の中長期の意味
5. **改善案**: 案を組み合わせる／改良する余地
6. **代替案**: 提示された案以外の第三の選択肢
7. **判断材料**: 選択のために人間が確認すべき情報

## Review Lens

- **目的妥当性**: 比較軸が目的に対して妥当か
- **範囲の過不足**: 重要な選択肢を見落としていないか
- **中長期リスク**: 短期最適が長期に不利にならないか
- **LAB全体との整合性**: LMS / 自動化 / B2B 展開と整合しているか
- **非エンジニア理解可能性**: 比較を関係者に説明できるか
- **他LLM移植耐性**: Claude 固有の解釈に依存していないか

## Instructions

1. 選択肢を列挙し、抜けている第三案がないか確認する
2. 比較軸を決める（コスト・効果・リスク・撤退容易性・前提依存）
3. 各案を同じ軸で評価し、表に落とす
4. 各案が前提とする「成立条件」を明示する
5. 撤退容易性（やめやすさ）を必ず軸に含める
6. 不明な評価は推測せず「未確認」と明示する

## Guardrails

- 結論を誘導する軸選びをしない
- 1案に有利な評価へ歪めない
- コスト比較（工数・機会コスト・撤退コスト）を省略しない
- 最終選択は人間に委ねる

## LAB Cross-Check

| 観点 | 状態 | 備考 |
|---|---|---|
| 自動化フロー | — | 各案の自動化前提・影響を比較したか |
| データ / 認証 / ログ | — | データ・権限面の差異を比較したか |
| 実装 / 運用フロー | — | 実装・運用負荷の差異を含めたか |
| 非エンジニア理解可能性 | — | 比較を関係者に説明できるか |
| 会員共有 / 再利用耐性 | — | 比較フレームが他案にも使えるか |
| 他LLM移植耐性 | — | 判断が Claude 固有に依存していないか |

状態は OK / 注意 / NG / 対象外 で記入すること。

## Handoff Notes

施工AI（Claude Code / Cursor 等）へ渡す前に以下を確定させること。

- **要件**: 採用案と、その採用理由
- **成功条件**: 採用案が機能したと判断する基準
- **失敗条件**: 別案へ切り替えるトリガー
- **実行範囲**: 採用案の実装で触れてよい範囲
- **影響範囲**: 採用が波及する領域
- **ロールバック方針**: 採用案が失敗した場合の戻し方・次善案
- **コスト比較**: 各案の工数・機会コスト概算
