## Purpose

価格と機能の消耗戦（正面衝突）に巻き込まれるリスクを避ける。
競合が追随しにくい独自の軸を見つけ、戦わずに勝てる位置取りを設計する。

## Use When

- 競合が多い市場で戦い方を決めるとき
- 「機能・価格で勝負」しか出ていないとき
- 差別化の軸が定まらないとき
- scope-design / value-proposition-check の後に独自性を固めたいとき

## Inputs

以下を準備すること。不足している場合は推測せず、不足を明示する。

- **対象市場**: 戦う市場・領域
- **主要競合**: 競合と、その強み・弱み（わかる範囲で）
- **自社の強み**: 模倣されにくい資産・能力
- **顧客の選定基準**: 顧客が選ぶときに見る軸

## Output Contract

以下の順で出力すること。順序を変えない。

1. **論点**: 正面衝突を避ける最有力の軸は何か
2. **根拠**: その論点をそう判断した理由
3. **ポジションマップ**: 競合と自社の位置（軸を明示）
4. **含意**: その独自ポジションの持続可能性・防御性
5. **改善案**: 差別化を強化・防御する打ち手
6. **代替案**: 別の差別化軸（チャネル・体験・ニッチ）
7. **判断材料**: ポジション確定に必要な人間の確認事項

## Review Lens

- **目的妥当性**: 差別化軸が顧客の選定基準に効くか
- **範囲の過不足**: ニッチすぎて市場が小さくないか
- **中長期リスク**: 模倣・追随で優位が消えないか
- **LAB全体との整合性**: LMS / 自動化 / B2B 展開と整合しているか
- **非エンジニア理解可能性**: 独自性を関係者に説明できるか
- **他LLM移植耐性**: 判断が Claude 固有の解釈に依存していないか

## Instructions

1. 顧客の選定軸を列挙し、競合が集中している軸を特定する
2. 競合が手薄／模倣しにくい軸を探す
3. 自社の強み（資産・能力）と結びつく軸を選ぶ
4. その軸での防御性（なぜ追随されにくいか）を言語化する
5. ニッチの市場規模が事業目標に足るかを確認する
6. 不明な競合情報は推測せず、要調査として明示する

## Guardrails

- 「競合より高機能」を差別化と混同しない（消耗戦の入口）
- 防御できない一時的優位を恒久優位と見なさない
- ニッチ化の市場規模リスクを省略しない
- ポジションの最終決定は人間に委ねる

## LAB Cross-Check

| 観点 | 状態 | 備考 |
|---|---|---|
| 自動化フロー | — | 差別化を支える自動化の有無を確認したか |
| データ / 認証 / ログ | — | 独自性を支えるデータ資産を確認したか |
| 実装 / 運用フロー | — | 差別化が実装・運用で再現できるか |
| 非エンジニア理解可能性 | — | 独自性を関係者に説明できるか |
| 会員共有 / 再利用耐性 | — | ポジション設計が他事業に転用できるか |
| 他LLM移植耐性 | — | 判断が Claude 固有に依存していないか |

状態は OK / 注意 / NG / 対象外 で記入すること。

## Handoff Notes

施工AI（Claude Code / Cursor 等）へ渡す前に以下を確定させること。

- **要件**: 確定した差別化軸と独自ポジション
- **成功条件**: 差別化が効いたと判断する基準
- **失敗条件**: ポジションを見直すトリガー
- **実行範囲**: 差別化を実装する範囲
- **影響範囲**: ポジション変更が波及する訴求・プロダクト
- **ロールバック方針**: 差別化が機能しない場合の戻し方
- **コスト比較**: 差別化を維持するコスト
