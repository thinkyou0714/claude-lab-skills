## Purpose

「始めたから続ける」サンクコスト的な惰性を防ぐ。
当初前提と実績のズレを直視し、継続・転換（ピボット）・撤退を冷静に選べる状態にする。

## Use When

- 施策・事業の定期レビュー（月次・四半期）
- 想定と違う結果が出てきたとき
- 「続けるべきか撤退すべきか」の判断が必要なとき
- goal-validation で立てた目標の進捗を査定するとき

## Inputs

以下を準備すること。不足している場合は推測せず、不足を明示する。

- **当初前提・目標**: 開始時に置いた前提と目標
- **実績**: 現時点の指標・観測結果
- **投下資源**: これまでの工数・費用（サンクコスト）
- **環境変化**: 開始後に変わった市場・前提

## Output Contract

以下の順で出力すること。順序を変えない。

1. **論点**: 継続判断を分ける最も重要な事実は何か
2. **根拠**: その論点をそう判断した理由
3. **前提-実績差分**: 当初前提 vs 実績のズレ（指標で）
4. **含意**: ズレが示す構造的な問題・好機
5. **改善案**: 継続する場合の修正（軌道修正）
6. **代替案**: 転換（ピボット）・撤退の選択肢と条件
7. **判断材料**: 継続/転換/撤退の判断に必要な人間の確認事項

## Review Lens

- **目的妥当性**: 当初目的が今も有効か（前提が崩れていないか）
- **範囲の過不足**: 査定対象が広すぎ／狭すぎないか
- **中長期リスク**: 惰性継続による機会損失リスク
- **LAB全体との整合性**: LMS / 自動化 / B2B 展開と整合しているか
- **非エンジニア理解可能性**: 査定結果を関係者に説明できるか
- **他LLM移植耐性**: 判断が Claude 固有の解釈に依存していないか

## Instructions

1. 当初前提・目標と実績を同じ指標で並べる
2. ズレの原因を「実行の問題」と「前提の崩れ」に切り分ける
3. サンクコストを判断から切り離す（これまでの投資は判断材料にしない）
4. 継続・転換・撤退それぞれの将来コストと期待値を整理する
5. 判断のトリガー（何が起きたらどう動くか）を明示する
6. 不明な実績は推測せず、計測の不足を明示する

## Guardrails

- サンクコストを継続理由にしない
- 「もう少し続ければ」の希望的観測を根拠にしない
- 撤退基準を曖昧にしない（事前に数値で）
- 継続/転換/撤退の最終判断は人間に委ねる

## LAB Cross-Check

| 観点 | 状態 | 備考 |
|---|---|---|
| 自動化フロー | — | 査定に必要な指標が自動取得できるか |
| データ / 認証 / ログ | — | 実績データの信頼性・取得経路を確認したか |
| 実装 / 運用フロー | — | 転換・撤退の実装影響を確認したか |
| 非エンジニア理解可能性 | — | 査定結果を関係者に説明できるか |
| 会員共有 / 再利用耐性 | — | 査定フレームが他施策に転用できるか |
| 他LLM移植耐性 | — | 判断が Claude 固有に依存していないか |

状態は OK / 注意 / NG / 対象外 で記入すること。

## Handoff Notes

施工AI（Claude Code / Cursor 等）へ渡す前に以下を確定させること。

- **要件**: 査定結果と、継続/転換/撤退の決定
- **成功条件**: 次レビューまでの判断トリガー
- **失敗条件**: 撤退・転換の発動基準
- **実行範囲**: 転換・撤退で触れてよい範囲
- **影響範囲**: 判断が波及する施策・顧客・チーム
- **ロールバック方針**: 転換が失敗した場合の戻し方
- **コスト比較**: 継続 vs 転換 vs 撤退の将来コスト
