# アンチパターン集 — lab-core 共通基盤

_過去に失敗または品質低下を招いた設計・判断の型。_
_同じ型を繰り返さないための参照リスト。_

---

## 設計系アンチパターン

### AP-D1: 単一巨大スキル化

- **症状**: 1つのスキルファイルが全領域をカバーしようとする
- **問題**: 責務が曖昧になり、再利用できなくなる
- **対処**: 1 Skill = 1責務。cross-plugin 参照は最小化

### AP-D2: AI固有フォルダを真実源にする

- **症状**: `.claude/` や `.cursor/` のファイルが唯一の仕様になっている
- **問題**: ツール変更時に知識が消える。移植不可
- **対処**: `src/` を SoT にして、AI向けは薄いアダプター層を作る

### AP-D3: 初手で過剰に作り込む

- **症状**: まだ使われていない機能・認証・DBを最初から本格実装する
- **問題**: 仕様変更に対応できず技術的負債になる
- **対処**: YAGNI（今必要なものだけ）。接続前提の空構造で留める

### AP-D4: README なしでファイルを増やす

- **症状**: ファイルだけ増えて、誰が何のために使うか不明
- **問題**: 新メンバーや未来の自分が読めない
- **対処**: 新ディレクトリには必ず README か用途コメントを置く

## 実装系アンチパターン

### AP-I1: 影響範囲の未確認

- **症状**: 変更する前に既存コードへの影響を確認しない
- **問題**: 本番コードが壊れる
- **対処**: 実装着手ゲート(GATE-1)を必ず通す

### AP-I2: ロールバック案なしの本番変更

- **症状**: 「戻せる」と思い込んで変更を進める
- **問題**: 問題発生時に復旧不能になる
- **対処**: 変更前にロールバック手順を文書化する

### AP-I3: 推測で仕様を埋める

- **症状**: 要件が不明確なまま実装を進め、後で手戻りが発生する
- **問題**: 手戻りコスト増大
- **対処**: 不足情報は推測せず明示して止まる

## 自動化系アンチパターン

### AP-A1: 全工程の完全自動化を初手で目指す

- **症状**: 手動確認を排除した完全自動パイプラインを最初に作る
- **問題**: 障害時に人間が介入できず、連鎖障害を起こす
- **対処**: 手動境界を明示し、最初は半自動から始める

### AP-A2: エラー通知なし自動化

- **症状**: 失敗してもサイレントで次へ進む
- **問題**: 問題の発見が遅れる
- **対処**: 失敗時には必ず通知・ログを残す設計にする

---

## 費用系アンチパターン

### AP-C1: API費用の無試算導入

- **症状**: 費用見積もりなしに外部API/SaaSを本番導入する
- **問題**: 費用が月次工数削減効果を超えた段階で気づく
- **対処**: 費用対効果ゲート(GATE-4)を通してから導入を決める
